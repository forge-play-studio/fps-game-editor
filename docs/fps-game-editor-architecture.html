fps-game-editor 架构当前状态与包设计

本文档是 fps-game-editor 新仓库的主架构文档，记录独立游戏编辑器生态的当前状态和已经确认的包设计共识。它不记录排期或任务拆分。当前主线是：平台传输意图，Vite 插件提供开发期注入，编辑器解释语义，项目适配 authored config，游戏运行时消费最终配置。新增核心原则是：永远以数据为准，Babylon Scene 只是某个 world 的渲染投影，不是权威数据源。平台对接原则是：平台负责承载、传输、调度、观测；编辑器负责解释、修改、保存、生成。当前工作原则是：快速迭代，只要方向正确即可继续推进，不急于过早优化或定死最终设计。

当前正式认知 forge-play、vite-plugins、lumber_order、pa_maker 都服务于新的独立游戏编辑器生态。文档主线不再以单个项目为中心，而是记录这些项目在当前架构里的实际职责。
平台对接红线 forge-play 可以优化 HostKernel、iframe/sandbox 生命周期、Service Worker proxy、资源传输、缓存、capture、性能观测和错误恢复；但 fps-game-editor 的 SceneDocument / EditorSession / AuthoringSource 才是编辑态权威。平台只能发送请求、传递资源和保存编辑器导出的结果，所有场景语义修改必须进入 editor command / project adapter / authoring source 链路，不能由平台 scene store、MCP tool、IndexedDB scene blob 或 sandbox runtime 直接写入权威场景数据。forge-play/docs/editor-game-api.md 当前只视为平台临时宿主协议草案，不能反向规定 fps-game-editor 的 source、保存或恢复语义。

迭代原则

当前工作原则我们不急于优化，也不急于把设计一次性定死。只要整体方向正确，就优先快速迭代、尽早接入真实项目，并用运行反馈修正边界。

  • 方向优先于完美抽象：先保持“平台传意图、编辑器解释、项目适配、运行时消费”这条主线成立。
  • 小步可回退：每轮迁移保持可验证、可回滚，不让未验证的抽象扩大影响面。
  • 事实推动设计：被真实接入、被验证、被文档化的能力，才逐步提升为稳定边界。
  • 不过早优化：性能、API 形状、包拆分细节在真实使用反馈出现后再收敛。

仓库定位

设计共识，部分实现本页记录已经讨论确认的包设计方向。当前仓库中 packages/editor-protocol、packages/editor-browser、packages/editor-forge-play、packages/editor-core、packages/editor-babylon、packages/editor 已有迁移实现；editor-browser 已提供最小 Local Editor UI shell，但还不是完整独立 editor-ui 产品层。本页内容不作为行动安排。

当前设计共识fps-game-editor 是独立游戏编辑器的代码仓库，不属于某一个游戏项目，也不属于 forge-play 平台内部实现。它服务于多游戏项目接入编辑能力，并通过 adapter 与具体项目交互。

编辑器运行在游戏 iframe 内，需要能访问浏览器环境、canvas、Babylon scene、项目 runtime、项目 authored config，同时通过 bridge 与平台通信。因此它不能是纯平台包，也不能是纯游戏业务包。

整体思路

当前设计共识当前设计不是把某个项目内的 editor-package 原样搬成公共包，而是把可复用的编辑器运行能力抽到独立仓库，把项目语义保留在项目 adapter 中。通用包负责“编辑器如何工作”，项目 adapter 负责“这个游戏的 authored config 应该如何变化”。

整体架构按职责分层，而不是按当前文件来源分层。平台只传递意图和保存结果；协议包定义共同语言；core 管命令、document 状态、生命周期和能力；browser/Babylon 包处理宿主环境和运行时观测；聚合包给游戏项目提供默认组装入口；项目 adapter 才理解 schema、资产规则和 runtime binding。

平台对接原则forge-play 可以提供更强的 HostKernel、sandbox 生命周期、Service Worker proxy、资源 bytes 管道、缓存、capture 和性能观测能力；但这些能力只属于宿主、传输、调度和观测层。平台不能把自己的 scene store、headless scene blob、MCP scene tools、IndexedDB restore、document:changed opaque snapshot 或 Babylon runtime object 升级为 fps-game-editor 的编辑态权威。平台侧临时 editor-game-api.md 中与此冲突的 document/restore 设计，应服从 fps-game-editor 的 authoring source 设计，而不是要求编辑器迁就平台 snapshot 模型。

层整体职责当前收敛方向
平台层承载预览、bridge、命令传输、文件保存、运行状态展示、sandbox 性能与可观测性。不理解项目 schema，不直接决定 authored config 的结构，不直接修改场景数据。
协议层定义命令、事件、capability 和跨包共享类型。保持轻量、稳定、向后兼容。
编辑器核心层解释通用编辑命令，维护 document 状态、history、dirty、生命周期和能力注册。只依赖抽象 adapter，不引入具体游戏规则。
运行时适配层处理浏览器宿主、Babylon scene、Inspector、gizmo、selection 和 runtime change 观测。产出 canonical change，把持久化交给项目 document adapter。
聚合入口层提供游戏项目默认接入入口，组合 core、browser、Babylon 和 Forge Play bridge。降低接入成本，但不吞掉项目 adapter 的语义职责。
项目 adapter 层实现 schema、资产注册、scene node 创建删除、runtime binding 和保存确认。保留项目差异，避免通用包绑定某个游戏。
游戏 runtime 层消费最终 authored config，创建真实运行时对象。只把配置当最终事实，不反向承担编辑器通用职责。

通用编辑器与状态化扩展

当前设计共识fps-game-editor 的核心能力应保持通用场景编辑，不把 PA 单帧制作里的状态化编辑直接塞进 core。状态化编辑应作为可选扩展建立在通用编辑能力之上，这样普通项目仍然可以只做标准场景编辑。

Unity 风格命名原则

当前设计共识核心数据和对外沟通命名可以偏向 Unity 风格，因为 Scene、GameObject、Component、Prefab、Asset、Material、Transform 对美术、技术美术和程序都更直观。但这只是术语和数据组织上的借鉴，不代表要复刻 Unity runtime、序列化系统或生命周期。

命名建议用法边界
SceneDocument 保存到文件或平台的完整场景文档容器，可包含 Scene、资源引用、预览快照和文档元信息。 不叫 EditorSceneDocument，因为核心数据不是编辑器私有；编辑器只是 author 它。
Scene 文档里的核心 authored scene 数据，包括对象层级、基础对象、组件和共享表现数据。 不是 Babylon Scene；Babylon Scene 只是某个 world 的 projection。
GameObject 替代长期对外的泛化 Scene Node 命名，表示文档里的 authored object identity。 不等于运行时 Babylon TransformNode，也不自动拥有 Unity 的 lifecycle。
Component 表达对象能力，例如 TransformComponent、ModelRendererComponent、DecalComponent、LightComponent、CameraComponent、OutlineComponent、GameplayBindingComponent。 用于拆开“对象身份”和“对象能力”，避免继续扩张 kind / transformType / overrides 大桶。
Prefab 编辑器层可复用对象模板，可展开为一组 GameObject + Component，并引用 Asset、Material 等资源。 第一版不引入复杂嵌套继承或 Unity Prefab Variant 语义。
Asset / Material / Transform 保持接近 Unity 的通用含义：原始资源、材质资源或参数、空间变换数据。 GLB 是 Asset，不等于 Prefab；Transform 是 authored data，不等于 Babylon runtime object。

因此命名方向是：用 Unity 风格降低沟通成本，用我们自己的数据边界保证 PA 编辑器链路清晰。不要引入 Awake、Start、Update 等 Unity lifecycle，也不要让 GameObject 成为某个具体 engine runtime object 的别名。

数据用途命名边界

当前设计共识命名必须先服务数据用途划分。不是所有可保存数据都属于 Scene，也不是所有运行时可调参数都应该回写 authored source of truth。

数据类别建议命名用途是否进入运行时
共享 authored 数据 SceneDocument / Scene / GameObject / Component / Prefab / Asset / Material 编辑器和运行时都需要的长期源数据。 通过项目 adapter 编译后进入 GameWorld。
编辑器私有数据 EditorWorkspace selection、展开折叠、editor camera、grid、snap、gizmo、panel layout、拖拽中间态。 不进入 GameWorld。
画面预览数据 PreviewSnapshot / SnapshotOverride 美术验收截图、关键画面或短视频状态；可以有时间和局部覆盖。 进入 PresentationPreviewWorld；不自动成为 gameplay state。
运行时 authored 配置 RuntimeTuning / GameConfig 程序或 AI 可调的运行参数，例如经济、AI 默认参数、机器参数、world bounds。 进入 GameWorld，但不是场景对象层级。
单次运行状态 RuntimeSessionState 玩家位置、NPC AI 状态、建造进度、资源数量、AI 动态调参即时值。 只属于一次运行；默认不回写 SceneDocument。

AI 在运行时做动态调整时，应先落在 RuntimeSessionState 或 RuntimeTuningDraft。只有经过人工确认或明确 apply/bake 流程后，才允许进入可发布的 RuntimeTuning 或 authored scene 数据。

Prefab 归属原则

当前设计共识Babylon 不提供 Unity 式完整 Prefab 系统；它提供的是 AssetContainer、clone、instance、thin instance、instantiateModelsToScene() 等渲染和实例化能力。最佳设计是由 fps-game-editor 定义编辑器层 Prefab 语义，Babylon 只作为 runtime projection / instantiate 后端。

概念定位说明
Asset 原始资源 例如 glb、texture、material、animation、vfx 文件。GLB 是资源，不等同于 Prefab。
Prefab authored template 编辑器里的可复用对象模板，可引用一个或多个 asset，包含默认 GameObject 层级、默认 component、默认 transform、默认材质和编辑器元数据。
PrefabInstance / GameObject 场景实例 场景中放置出来的一个 authored object，引用 prefabId 或直接挂载 components，拥有自身 transform 和 instance override。
Babylon objects 渲染投影 Mesh、TransformNode、Material、Light 等是 Prefab/GameObject 在某个 world 中的投影，不是 Prefab 本身。

因此 Prefab 应是通用编辑器能力，不只属于 PA 单帧制作。PA 的画面快照和状态化编辑建立在 Prefab / GameObject 之上。

Asset
  -> Prefab

当前实现状态 当前已落地的是 scene.main 保存主链路：公共框架提供 AuthoringCommand、AuthoringSourceDriver、ProjectAuthoringHost、diagnostics、revision check、runtime patch 类型和 RuntimeAdapter 接口；LocalEditorHarness.saveScene() 已优先调用 ProjectAuthoringHost.commitSource(...)。尚未完整落地的是项目级常驻 host、AI / RuntimeWorld 统一 dispatch(...) 接入、项目侧 compileRuntimePatch 实现，以及实际 RuntimeAdapter 挂载和运行时 apply。

坐标系约定

当前设计共识fps-game-editor 的 Babylon world projection 必须和项目 GameWorld 使用同一套坐标系。当前 lumber_order 基线使用右手坐标系，GameWorld 设置 scene.useRightHandedSystem = true，因此本地 EditorWorld 也必须以右手坐标系创建。

选择右手坐标系的原因不是单纯技术偏好，而是为了贴近美术生产流程：美术导出的 GLB 资源默认是右手坐标系，编辑器保持右手系可以让美术在查看原始资源、摆放资产和进入运行时验证时看到一致的方向关系，减少坐标换算和方向误判。

  • SceneDocument.transform.position / rotation 是 authored data，不因进入不同 world 而改写语义。
  • EditorWorld、PresentationPreviewWorld 和 GameWorld 的 Babylon Scene 必须显式声明同一坐标系；不得在某个 projection 层临时翻转坐标作为长期方案。
  • 如果后续接入其他项目，该项目 adapter 必须声明自己的 world 坐标系；默认应优先保持 authoring data 与 runtime input 坐标系一致。

数据权威原则

当前设计共识编辑器需要借鉴帧同步和现代引擎的分层思想：逻辑层和渲染层分离，数据是权威，Babylon runtime object 只是数据在某个 world 中的表现。

当前收敛PresentationPreset 暂时不作为第一版核心概念。第一版用 PreviewSnapshot 直接记录相对 Base Scene 的差异即可，避免为了复用过早增加一层抽象。

预览首帧 / 表现预览

当前设计共识“预览首帧”是 PA 制作流程里的工作叫法，不代表内部固定到某一帧。更准确的工程概念是 PresentationPreview：从当前 authored data 快照重建出来的只读视觉验证 world，可用于截图、关键状态画面或短视频预览。

预览能力边界
可以选择某个画面快照，例如营地未建成、营地建造中、营地建成后、怪物登场、结算画面。 这些快照是 authored visual snapshot，不是 gameplay runtime state。
可以启用接近最终效果的灯光、阴影、描边、后处理、相机、角色 pose、粒子、动画采样和必要 visual warmup。 不运行完整 gameplay，不启用玩家输入、AI、经济、战斗、任务推进。
可以包含时间概念，例如 previewTime、duration、warmupFrames、loopAnimation，用于输出某个状态截图或短视频。 时间只服务视觉预览，不代表 gameplay 时间线已经真实推进。
可以基于未保存的 working copy 生成 dirty preview，方便快速迭代。 预览退出后销毁 preview world，不把预览中的 runtime 变化写回文档。

运行游戏

概念边界，暂不展开实现运行游戏属于完整 GameWorld。当前文档只固定它与编辑/预览的边界：程序逻辑状态是运行时权威，美术 authored data 提供初始摆放和人工参考，运行时变化默认不回写 SceneDocument。

  • 运行游戏按钮用于验证玩法，不用于编辑 authored document。

美术 visualState 与程序逻辑状态

当前设计共识美术定义的画面状态不能成为程序逻辑的权威状态。当前第一版推荐以 PreviewSnapshot 和 SnapshotOverride 描述这些画面状态；PresentationPreset 只作为未来可能的复用抽象。

概念由谁定义用途不是
PreviewSnapshot / 画面快照 美术 / 编辑器 定义某个截图或视频预览状态下场景应该如何呈现，例如营地未建成、建造中、建成后。 不是程序逻辑状态，不证明游戏里对象已经真实进入该状态。

EditorSession 状态管理初版方案

当前实现状态 EditorSession、Undo/Redo、Dirty/History 和 projection sync 主链路已经收口到 @fps-games/editor-core / @fps-games/editor；Hierarchy、Inspector、Asset Browser、Gizmo/View Plane Move 等入口已经进入同一套 document command 体系。后续继续补齐的是 GameObject 复制、Hierarchy before/after 排序、Prefab、PreviewSnapshot、完整 Material/Light/Camera/Decal source、runtime apply、AI 动态调参边界和正式 GameWorldInput。lumber_order 只是第一个接入项目，不是这套框架的归属地。

总原则

Hierarchy / Inspector / Gizmo / Menu Command

  -> Serialized SceneDocument / Serialized Object
  -> Undo Transaction
  -> Dirty Mark
  -> Projection Sync
  • SceneDocument working copy 是编辑态唯一权威数据源。
  • Hierarchy、Inspector、Gizmo、菜单命令只是不同输入源，不能各自直接改 JSON 或 Babylon object。
  • Babylon EditorWorld 只做 projection、拾取、可视化和交互输入；不能成为保存数据的来源。
  • 所有会修改 authored data 的操作必须进入 Undo transaction，并标记 dirty。
  • Selection、面板展开、filter、hover、projection loading 等 UI/session 状态不进入 Dirty，也不进入保存文件。

Redux 可借鉴但不照搬

Redux 思想对编辑器的映射采用方式
Single Source of Truth WorkingDocument 采用。Babylon Scene、UI 面板和 Inspector 都不是权威数据源。

Dispatch 流程

editorSession.dispatch(command)
  -> validate lifecycle allows command
  -> begin or join transaction if DocumentCommand
  -> reduce WorkingDocument / SerializedObject
  -> reconcile SelectionState
  -> mark dirty if document changed
  -> push or update UndoStack transaction
  -> notify projector to sync Babylon objects
  -> notify UI to render derived state

当前设计共识新 Gizmo 系统属于独立 EditorWorld 主链路，不复用旧 runtime path 的 tool-controller.ts。Babylon GizmoManager 只作为 3D 控件渲染、拾取和拖拽输入后端；权威数据仍然是 SceneDocument working copy。

@fps-games/editor-core 定义 transform tool、space、snapshot、批量 transform commit、pivot 和 constraint 等无引擎类型；事务仍通过 EditorSession.dispatch(document.patch) 进入 undo/dirty。 不依赖 Babylon，不包含项目 schema，不实现项目 GameObject/Prefab 规则。

@fps-games/editor Local harness 连接 selection、projection、gizmo event、View Plane Move 和 EditorSession，一次 drag end 提交一条 document patch。 不从 Babylon scene 反扫生成 document。
project adapter 把单个或批量 transform input 转成项目 schema patch；例如 lumber_order 更新 GameObject Transform component 并 compile 到 scene.json。 不实现通用 gizmo 状态、多选 pivot、View Plane 算法，不直接操作 Babylon GizmoManager。

View Plane Move 是 Move 工具的一种显式 constraint。它以当前 editor camera 方向构造穿过 pivot 的拖拽平面，鼠标 ray 与该平面求交得到位移 delta。该能力只影响 preview 和最终 commit，不改变保存链路，也不允许从 Babylon Scene 反扫 authored data。

Multi Inspector 精确编辑

当前设计共识多选 Inspector 必须消费序列化 read model，而不是读取 Babylon object。多选 Transform 字段支持 mixed value：全部对象一致时显示数值，不一致时显示 --。用户输入数值后，对所有 selected objects 设置同一个绝对值，并通过项目 adapter 生成批量 transform patch。

Rotation 在 UI 中使用 degrees，方便美术按 Unity 心智输入；document 与 Babylon projection 仍使用 radians。degrees/radians 转换属于项目 adapter 和 serialized descriptor 的边界，不应污染 editor-core 的通用数据模型。

Gizmo 事务模型

第一版使用 preview-then-commit 模型：拖动中只移动 projection 作为视觉预览，松手时提交一次 DocumentCommand。这样当前 EditorSession 的 snapshot undo 模型不需要为每帧 patch 增加 transaction merge。

gizmo.dragStart
  -> capture before transform from selected projection root

gizmo.dragMove
  -> Babylon GizmoManager moves projection root
  -> no document mutation
  -> no undo transaction

gizmo.dragEnd
  -> read after transform from projection root
  -> create project transform patch
  -> editorSession.dispatch(document.patch)
  -> push one undo transaction
  -> sync projection from WorkingDocument

gizmo.dragCancel
  -> restore projection root to before transform
  -> no document mutation
  -> no dirty change

当前确认边界

forge-play 联调边界 平台优化应围绕“承载、传输、调度、观测”展开。编辑器可以对平台暴露稳定接口，但平台不得直接修改 SceneDocument / EditorSession / AuthoringSource、不得把 Babylon runtime scene 当成 authored source of truth、不得用平台侧 scene store 或浏览器 IndexedDB restore 覆盖项目 adapter / source driver 的保存链路。forge-play/docs/editor-game-api.md 当前只能作为临时平台宿主协议草案；其中与 fps-game-editor 数据权威冲突的部分必须调整。
边界当前状态原因
平台宿主能力与编辑器数据权威 平台可优化 HostKernel、sandbox 生命周期、Service Worker proxy、资源传输、缓存、capture、性能观测和错误恢复；编辑器保留全部场景语义修改权。 这些平台能力提升运行和联调体验，但不应该改变 SceneDocument / EditorSession / project adapter 的权威位置。
平台与项目 schema 平台侧不应理解项目 scene.json 结构。 不同项目可能拥有不同 authored schema；平台理解 schema 会形成跨项目耦合。
平台与用户操作 平台侧负责识别用户操作并发送意图消息。 平台知道拖拽、点击、文件路径、保存请求，但不应知道这些动作如何变成项目节点。
编辑器与项目语义 编辑器侧负责把平台意图解释为项目编辑语义。 编辑器 runtime 维护工作副本、history、selection、Inspector 交互。
项目 adapter 与 authored config 项目 adapter 负责更新 authored config。 scene.assets[]、scene.nodes[]、材质、outline、transform 等字段属于项目规则。
平台 scene store / MCP tools 与编辑器命令 平台侧 scene store、headless scene blob、MCP scene tools 不能直接写 fps 项目场景；如果需要自动化编辑，必须调用编辑器暴露的 command API。 自动化工具可以是调用方，但不能成为数据所有者；否则会绕过 undo/redo、dirty、selection、schema 校验和项目 adapter。
editor-game-api.md 与 fps editor 数据协议 该文档中的 lifecycle:*、mode:*、capture:*、assets:*、system:* 可作为 Host Transport Contract 候选；document:changed、document:restore、window.__sandbox.host.scene 这类 opaque scene snapshot/restore 机制不能作为 fps editor 的权威数据协议。 前者属于宿主承载与传输，后者会让平台缓存或运行时 snapshot 绕过 AuthoringSource(scene.main)、revision check、validate/save/compile 和项目 source driver。
IndexedDB / sandbox restore 与正式保存 平台缓存只能作为宿主级性能优化或非权威恢复辅助；对 fps 项目不得自动把 IndexedDB 中的 scene blob restore 到 SceneDocument 或运行时并视为保存事实。 缓存 blob 可能来自运行时投影、旧协议或过期页面状态；正式保存必须来自 ProjectAuthoringHost.commitSource(...)，并通过项目 source driver 校验、保存和编译。
游戏 runtime 与配置消费 游戏 runtime 只消费最终配置。 游戏运行态不应该依赖平台保存过程；刷新后由配置回放。
SceneBuilder 归属 SceneBuilder 是项目/游戏侧能力，不是平台侧能力。 它读取项目 config 并创建 Babylon runtime node，天然绑定项目 schema 和游戏服务。
当前正式链路 平台发送的是 asset.import、mode.change、document.export、document.commit 这类命令或保存动作；项目侧决定如何修改 scene.json。

历史背景与临时绕行

历史背景本页只保留此前讨论中仍有解释价值的背景。它们不构成当前正式架构主线。

资产拖拽与 legacy 格式错配

此前观察到平台 bridge 可能把导入资产写入 sceneInstances 或旧 persistence payload，而 lumber_order 的 SceneBuilder 只读取 Scene JSON V2 的 scene.assets[] + scene.nodes[]。这会导致“运行时看到资产，但保存或刷新后不稳定”的问题。

当前正式链路平台只发意图，项目 editor runtime 写 V2。

Mock Platform Assets

Mock Platform Assets 是项目侧模拟平台流程的调试面板。它用于验证“平台意图消息 -> 项目 editor runtime -> Scene JSON V2 -> 保存 -> 刷新回放”的闭环，不代表 forge-play 必须实现同一套 UI。

AssetManager 职责收敛

当前共识是：AssetManager 管资产注册和资产引用，不管理场景 instance 或场景树。场景 instance 属于 editor document / scene document，运行时实体由 SceneBuilder 创建。

当前收口asset.import 返回成功必须同时代表 Scene JSON V2 working copy 已写入、runtime node 已创建。runtime 创建失败时项目 adapter 回滚本次 document 变更并返回失败。

通用传输策略
