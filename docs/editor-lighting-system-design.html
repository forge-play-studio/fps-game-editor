fps-game-editor design document

编辑器光照系统设计

本文档记录 fps-game-editor 光照系统的产品语义、工程边界和 BabylonJS 适配方式。 核心目标是：编辑器作为光照作者态真相源，运行时只消费编辑器导出的稳定光照配置； 同时利用 BabylonJS 的渲染能力，但不让编辑器核心模型过度耦合 BabylonJS 对象模型。

设计摘要

光照系统应以 fps-game-editor 自有的场景文档和组件语义为中心。BabylonJS 是当前主要渲染后端， 负责把编辑器光照模型投影成真实的 HemisphericLight、DirectionalLight 和相关渲染状态， 但它不应该成为编辑器文档、Inspector Schema 或导出协议的权威模型。

核心判断 理想情况下，编辑器预览与运行时可以做到视觉一致。前提是光源、背景、渲染管线、材质和相机都来自同一份作者态参数， 而不是 editor、runtime、pa_template 分别维护各自的默认值。

编辑器

作者态真相源

负责创建、保护、编辑和导出光照参数。Inspector 修改的是 scene document，而不是 Babylon runtime object。

editor-babylon

渲染适配层

负责把 editor domain model 映射到 BabylonJS lights、颜色、向量、sky、grid 和 preview 状态。

runtime

导出配置消费者

只读取编辑器导出的 scene/runtime config，不重新发明光照规则，不维护隐藏兜底光源。

边界原则

  1. 编辑器模型优先。核心文档里保存的是编辑器认可的稳定光照语义，不保存 BabylonJS class instance。
  2. 命名可以对齐 BabylonJS，协议不被 BabylonJS 绑死。UI 可以显示 Hemispheric Light、Directional Light，但核心类型仍然是编辑器自己的 light component。
  3. 两个 protected light 是当前 PA 游戏的默认光照面。现阶段只保留环境光/半球光与直射光，不暴露阴影相关属性。
  4. 编辑器辅助层不是游戏内容。grid、gizmo、selection outline 可以随光照调整可读性，但不进入 runtime 光照语义。
  5. 同源默认值。光照默认值不能散落在 editor world、projection、pa_template runtime、rendering.json 多处。
  6. 运行时只消费导出结果。runtime 可以使用 BabylonJS 创建灯光，但参数必须来自导出的 scene config 或稳定 fallback resolver。

作者态模型

当前建议把光照作者态收敛到两个 protected scene object。它们是场景的系统节点，可以在 Hierarchy 中看到，可以在 Inspector 中编辑有限字段， 但不能被普通删除、改成其他组件或降级为普通 transform。

系统 ID 显示名称 编辑器类型 BabylonJS 映射 当前暴露字段
environment_light Hemispheric Light light.type = "hemispheric" BABYLON.HemisphericLight enabled、intensity、diffuseColor / sky light color、groundColor
sun_light Directional Light light.type = "directional" BABYLON.DirectionalLight enabled、intensity、direction、diffuseColor

建议的中间模型

type EditorLightKind = "hemispheric" | "directional";

interface EditorColorRGB {
  r: number;
  g: number;
  b: number;
}

interface EditorVector3 {
  x: number;
  y: number;
  z: number;
}

interface EditorHemisphericLight {
  type: "hemispheric";
  enabled: boolean;
  intensity: number;
  diffuseColor: EditorColorRGB;
  groundColor: EditorColorRGB;
}

interface EditorDirectionalLight {
  type: "directional";
  enabled: boolean;
  intensity: number;
  direction: EditorVector3;
  diffuseColor: EditorColorRGB;
}

type EditorLight = EditorHemisphericLight | EditorDirectionalLight;
命名说明 diffuseColor 更贴近 BabylonJS 的字段；中文 Inspector 可显示为“天空光颜色”。 如果产品语义希望更明确，也可以在 editor domain 中使用 skyLightColor，由 Babylon adapter 映射到 diffuse。

Lighting Resolver 是建议新增的关键边界。它不依赖 BabylonJS，只接收 editor scene document 和默认值， 输出稳定的 SceneLightingProfile。editor-babylon、导出器、pa_template runtime 都可以围绕这个 profile 对齐行为。

编辑器预览

编辑器预览应分成“游戏内容受光”和“编辑器辅助可读性”两个层面。mesh、primitive、模型材质应该真实受光； 背景、sky、grid、gizmo、outline 则需要显式跟随 lighting profile 调整，而不是期待 BabylonJS light 自动影响它们。

预览对象应如何受光照参数影响是否导出到 runtime
游戏 mesh / primitive 使用真实 Babylon light 参与材质计算。 是，作为场景内容。
scene clearColor / background 由 lighting profile 显式派生或由 scene environment config 指定。 建议导出，保证 editor/runtime 一致。
editor sky backdrop 如果存在，应读取同一份 sky/background profile；不能固定为独立编辑器风格。 视产品决定。若 runtime 也有 sky，应导出。
editor grid 根据环境亮度调整颜色和透明度，保持可读。 否，只是编辑器辅助层。
gizmo / selection outline 保持可读性，必要时做亮暗主题适配。 否，只是编辑器辅助层。

运行时消费

运行时的职责是消费导出后的光照参数，并创建对应的渲染对象。它可以继续使用 BabylonJS， 但不应该在 SceneBuilder、RenderingService 或 rendering.json 中维护一套和编辑器无关的光照默认规则。

目标状态 同一份 exported scene config 在编辑器和 runtime 中产生同一组主要光源、主要背景/环境色和主要 image processing 参数。

运行时可以保留的能力

  • 把导出配置转为 new HemisphericLight(...) 和 new DirectionalLight(...)。
  • 根据导出配置应用 scene.clearColor、曝光、对比度、bloom 等渲染管线参数。
  • 在缺少旧项目配置时使用 resolver 提供的兼容默认值。

运行时应避免的行为

  • 在 scene config 已有 protected lights 时继续创建隐藏兜底灯。
  • 使用独立于编辑器的 hard-coded clearColor、light intensity 或 direction。
  • 把 shadow、IBL、sky 等未来能力隐式打开，导致编辑器看不到但运行时生效。

BabylonJS 适配边界

BabylonJS 相关 API 应集中在 packages/editor-babylon 和项目 runtime 的 Babylon adapter 中。 editor core、protocol、document、Inspector schema 不应该直接依赖 Color3、Vector3、HemisphericLight 或 DirectionalLight。

编辑器语义BabylonJS 映射边界说明
{ r, g, b } new Color3(r, g, b) 转换只发生在 Babylon adapter 内。
{ x, y, z } new Vector3(x, y, z) 核心协议只使用 plain object。
type: "hemispheric" HemisphericLight UI 名称可显示为 Hemispheric Light，但存储类型不绑定 class name。
type: "directional" DirectionalLight 方向、强度、颜色由 profile 归一化后传入。

Inspector 设计

Light Inspector 应延续 Camera Inspector 的经验：中文/英文属性名可切换，字段有 tooltip，原始数据可只读查看。 Inspector 展示的是编辑器认可的属性子集，而不是 BabylonJS 支持的全部属性。

字段中文名英文名说明
language语言LanguageInspector 自身显示语言，不进入 runtime export。
type类型Typeprotected light 的类型只读。
intensity强度Intensity非负数。建议保留可输入形式，后续可增加 slider。
diffuseColor天空光颜色 / 光照颜色Sky Light Color / Light Color半球光对应 sky light，直射光对应 diffuse light color。
groundColor地面颜色Ground Color仅半球光暴露。
direction方向Direction仅直射光暴露。
raw原始光源数据Raw Light Data只读，辅助调试。
暂不暴露 shadow、specular、range、falloff、include/exclude meshes、lightmap、IBL 等 BabylonJS 能力暂不进入首版 Inspector。

当前代码观察

本节记录当前仓库里和设计相关的观察点，帮助后续实现时定位需要收敛的地方。

区域当前现象设计影响
.local/pa_template/src/services/SceneBuilder.ts 运行时已经创建 HemisphericLight 和 DirectionalLight。 说明 runtime 有基础，但需要改成消费编辑器导出的同源参数。
.local/pa_template/src/config/rendering.json 存在 globalVolume.lights、exposure、contrast、bloom、shadow 配置。 可作为兼容 fallback，但不应长期成为编辑器 protected lights 之外的另一套作者态。
.local/pa_template/src/config/types.ts 当前 scene light 类型主要覆盖 directional。 需要补齐 hemispheric/ambient light 的 scene model。
.local/pa_template/src/core/Game.ts runtime scene.clearColor 是 hard-coded。 若要求 editor/runtime 背景一致，需要纳入 lighting/environment profile。
packages/editor-babylon/src/editor-world.ts editor world 存在默认 editor-world-light 兜底。 应清理隐藏兜底，避免 protected lights 失去作者态权威。
packages/editor-babylon/src/editor-sky.ts editor sky material 使用 disableLighting = true。 sky 不会自动受灯光影响，需要 profile 显式驱动颜色。

实现建议

  1. 在 editor domain 中定义稳定的 EditorLight / SceneLightingProfile，不要让 core 依赖 BabylonJS 类型。
  2. 确保 scene document 默认存在两个 protected lights：environment_light 和 sun_light。
  3. 新增或收敛一个 resolver，统一处理默认值、数值 clamp、颜色 fallback、方向归一化和 preview 派生参数。
  4. 让 editor-babylon 只负责将 profile 映射到 Babylon lights、clearColor、sky tint 和 grid contrast。
  5. 导出器从同一份 profile 或 scene document 生成 runtime config，剥离 Inspector-only 字段。
  6. pa_template runtime 从导出的 scene config 读取两盏灯；旧 rendering.json 光照配置只作为缺省兼容。
  7. 为 editor preview 与 runtime 增加同源配置测试，至少覆盖 light intensity、color、direction、clearColor。

验收清单

  • Hierarchy 中固定存在 protected Hemispheric Light 与 Directional Light。
  • 关闭或调暗 protected lights 后，游戏内容不会被隐藏兜底光继续照亮。
  • 半球光强度、天空光颜色、地面颜色会影响真实受光材质。
  • 直射光强度、方向、颜色会影响真实受光材质。
  • 背景/sky/grid 的视觉变化来自 lighting profile 的显式派生，而不是随机硬编码。
  • Inspector 支持中文/英文切换和 tooltip，字段子集清晰，不暴露 shadow。
  • 导出的 scene config 不包含 inspectorLanguage 等编辑器 UI 字段。
  • runtime 与 editor 在相同 scene config 下创建同一组主要光照参数。
  • editor core/protocol 不引用 BabylonJS runtime class。

待确认问题

  • 半球光在 editor domain 中字段名使用 diffuseColor 还是 skyLightColor。
  • 背景色是否作为光照 profile 的派生结果，还是作为独立 SceneEnvironment 作者态字段。
  • runtime 是否需要与 editor 使用同一套 sky/backdrop，还是第一阶段只对齐 clearColor。
  • rendering.json 的 globalVolume.lights 是否长期保留，还是逐步迁移为 scene document 默认值。
  • 阴影能力何时进入 Inspector，以及是否作为 directional light 子属性还是独立 shadow profile。

本文档用于记录当前设计共识。后续实现应以编辑器作者态模型为中心，逐步把 BabylonJS 细节收敛到 adapter 层。
