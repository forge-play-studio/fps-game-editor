forge-play / lumber_order / fps-game-editor 本地联调方案

本文档记录 2026-05-19 确认的本地联调环境、启动顺序、登录绕过方式、验证点和已知风险。目标是先验证 forge-play HostKernel 能加载本地 lumber_order，并让 lumber_order 使用联调版 fps-game-editor 源码；第一阶段只在项目侧接入平台现有 mode:change 命令，协议完全打通是下一步工作。

重要边界 本轮本地联调不走真实创建项目、E2B sandbox、GitHub 发布或飞书 OAuth。不要在首页弹窗里点击“加入项目”或“从模板创建”。联调入口使用 previewUrl 直接把本地 Vite 页面放进 forge-play 的 sandbox iframe。

三、首次准备

新 worktree 通常没有 node_modules，首次启动前执行一次即可。

cd /Users/admin/work/UGIT-Branch/lumber_order-forge-play

如果 forge-play 已经安装过依赖，也可以先跳过；但当前联调分支新增了 MCP server 依赖，缺 mitt 或 zundo 时必须补装。

四、启动顺序

1. 构建 bridge 与 MCP server

cd /Users/admin/work/UGIT/forge-play
pnpm --filter @repo/mcp-bridge build
pnpm --filter @repo/mcp-server build

2. 启动 8080 bridge/MCP 服务

cd /Users/admin/work/UGIT/forge-play
pnpm --filter @repo/mcp-server dev

期望看到：

[Server] http://0.0.0.0:8080
[CLI] E2B_SANDBOX_ID not set — skipping wiki bootstrap (local dev)

3. 启动 lumber_order 本地游戏

cd /Users/admin/work/UGIT-Branch/lumber_order-forge-play
pnpm run dev:editor-local

dev:editor-local 已固定为 --host 0.0.0.0 --port 5183 --strictPort，避免和其他本地 Vite 服务抢 5173。如果 5183 被占用，Vite 会直接失败，先释放端口后再启动。

4. 启动 forge-play Web

cd /Users/admin/work/UGIT/forge-play
pnpm --filter web dev

通常只启动 web 即可。如果 Web 因 agent 构建产物缺失而报错，再改用完整启动：

cd /Users/admin/work/UGIT/forge-play
pnpm run dev

五、打开联调页面

浏览器打开：

http://localhost:3000/zh/workspace/local-lumber?previewUrl=http%3A%2F%2Flocalhost%3A5183

local-lumber 是本地烟测用的假 sandboxId。因为 URL 里带了 previewUrl，forge-play 的 Editor 会跳过真实 E2B sandbox polling，直接把 iframe 指向 http://localhost:5183。

六、登录绕过

forge-play 的 middleware 会要求 fp_session cookie。由于本轮只做本地 previewUrl 烟测，可以在浏览器 Console 临时写入 cookie：

document.cookie = "fp_session=local-dev; path=/; SameSite=Lax";
location.href = "/zh/workspace/local-lumber?previewUrl=http%3A%2F%2Flocalhost%3A5183";
限制 这只是在本地绕过页面门禁，不代表真实登录。创建项目、加入项目、发布、GitHub、E2B、飞书 OAuth 仍然需要正式登录和完整环境变量。

七、健康检查

curl -I http://localhost:8080/script/bridge.js
curl -I http://localhost:5183/
curl -I -H 'Cookie: fp_session=local-dev' 'http://localhost:3000/zh/workspace/local-lumber?previewUrl=http://localhost:5183'
检查项期望说明
8080/script/bridge.js 200 OK 证明 lumber_order 能加载平台 bridge 脚本。
5183/ 200 OK 证明本地游戏 Vite 服务可用。
带 cookie 访问 workspace URL 200 OK 证明 forge-play Web 能进入项目页；不带 cookie 会被重定向到登录页。

八、第一轮验证目标

  • forge-play workspace 页面能进入 local-lumber 项目页。
  • HostKernel 能把 http://localhost:5183 加载进 iframe。
  • lumber_order 页面能正常渲染游戏画面。
  • lumber_order 的 Vite alias 确实指向 fps-game-editor-forge-play。
  • bridge.js 能注入，至少具备本地 bridge 基础能力。
  • forge-play 现有 mode:change 命令进入 iframe 后，项目侧能进入 EditorWorld。
当前阶段 当前只打通“forge-play 现有 mode:change 命令到项目侧 EditorWorld 切换”。本阶段不要求修改平台编辑按钮或 inspector 交互逻辑；无论命令来自平台按钮，还是来自平台当前分支已有的 ready 后自动进入编辑态逻辑，项目侧都只把它当作授权的模式切换命令。本阶段暂不处理场景保存、恢复、资产落盘、协同同步和发布提交的数据流。
mock API 退出 __mock_platform_assets 是早期为了本地快速模拟平台能力的临时方案。进入 forge-play 真实 HostKernel / sandbox 联调后，编辑器运行时不应再依赖这些 dev-only HTTP API。第一阶段改为：编辑器初始文档和素材库由 lumber_order 项目自身 adapter 提供；平台侧只发送模式切换命令，保存/恢复/资产写入暂不接入。

九、已知限制与下一步

协议未完全对齐 页面能加载不代表编辑协议已经完成。当前 forge-play 分支 Host 侧使用 mode:change、document:export、document:commit、assets:drop；而 fps-game-editor / lumber_order 当前主协议仍是 mode.change、document.export、document.commit、asset.import。

后续需要补的平台兼容层：

平台新协议编辑器当前协议处理建议
mode:changeLocalEditorHarness.enterEditor()第一阶段只做模式切换：项目侧接收平台既有命令后，授权进入 EditorWorld，并回发 mode:changed。
document:exportdocument.exportadapter 转发，并把 document.exported 回包转成平台期望。
document:commitdocument.commitadapter 转发给项目 document adapter。
assets:dropasset.import / 项目 asset adapter平台只传资源与落点，场景修改由编辑器和项目 adapter 决定。
register runtimeregisterEditorRuntime / registerEditorPluginforge-play bridge 需要恢复兼容入口，或提供等价注册 API。

第一阶段实现约束

  • lumber_order iframe 就绪后主动发送 lifecycle:ready 和当前 mode:changed。
  • 本阶段不改 forge-play 的按钮或 inspector 行为，只适配它当前已经发送的 mode:change 命令。
  • lumber_order 只监听 mode:change，并调用本项目授权的 LocalEditorHarness.enterEditor()。
  • 本阶段保存、恢复、资产导入仍保持未接入状态，避免平台误认为它已经拥有场景数据更新权。

十、排障表

现象原因处理
打开项目页又回到登录弹窗 没有 fp_session cookie。 执行“登录绕过”里的 Console 代码，或正式登录。
首页出现“创建新项目”弹窗 进入了真实 workspace 创建流程。 关闭弹窗，直接打开带 previewUrl 的联调 URL。
Port 5183 is already in use 联调端口已被其他本地服务占用。 先释放 5183，再运行 pnpm run dev:editor-local。脚本带 --strictPort，不会静默漂移到其他端口。
MCP server build 缺 mitt 或 zundo forge-play 当前分支新增依赖，但根目录没安装。 在 /Users/admin/work/UGIT/forge-play 执行 pnpm install。
Next Build Error：Can't resolve '@repo/agent/server/commit-message' apps/web/tsconfig.json 的通用 @repo/* path 会把深层导入错误解析到不存在的 packages/agent/server/commit-message/src。 为 @repo/agent/server/commit-message 增加精确 path，指向 packages/agent/server/commit-message.ts。
修完后又出现 Can't resolve '@agent/prompt' 或 @pierre/diffs/worker 把 @repo/agent/client 指到源码会让 Next 编译 agent client 的 Vite 专用代码。 @repo/agent/client 应指向 packages/agent/dist/client/index.mjs，@repo/agent/styles.css 指向 dist/client/assets/agent.css；server commit-message 才指源码。
bridge.js 404 或加载失败 8080 服务没启动，或 bridge/server 未构建。 重新执行 bridge 与 MCP server 构建，再启动 pnpm --filter @repo/mcp-server dev。
页面可见但 Edit / Save / Asset Drop 不工作 平台新协议和 fps-game-editor 当前协议尚未适配。 进入下一步：补协议 adapter，不能让平台直接修改权威场景数据。
更新 forge-play / Service Worker 后，Save & Exit 点击无反应 浏览器仍可能被旧的 Service Worker 控制。典型判断是：点击后 forge-play 和 lumber_order 终端都没有出现 /__fps_editor_authoring/save-editor-scene 请求日志，这说明命令还没有进入新的保存链路。 在当前页面 Console 执行： 
navigator.serviceWorker.getRegistrations()
  .then(rs => Promise.all(rs.map(r => r.unregister())))
  .then(() => location.reload());
这会注销当前站点旧的 Service Worker 并刷新页面，让浏览器重新加载最新的 forge-play 前端和 sandbox proxy 逻辑。 正常联调不需要每次执行；只有改过 apps/web/public/sw.js、proxy、iframe 注入或 Save & Exit 链路后行为仍像旧版本时使用。

十一、停止服务

优先在对应终端里按 Ctrl+C 停止：

  • pnpm --filter @repo/mcp-server dev
  • pnpm run dev:editor-local
  • pnpm --filter web dev

如果终端丢失，可以再按端口查进程并手动停止。

十二、核心原则

数据权威 平台负责承载、传输、调度、观测；编辑器负责解释、修改、保存、生成。forge-play 可以优化 HostKernel、iframe/sandbox 生命周期、Service Worker proxy、资源传输、缓存、capture、性能观测和错误恢复；但 SceneDocument / EditorSession / project adapter 仍然是编辑态权威链路。
侵入边界 平台 API 不能被设计成“平台拥有一份可编辑的通用 scene graph，并直接 patch 节点、材质、transform 或资产实例”。平台可以保存、恢复、传输编辑器导出的 opaque document，也可以向编辑器发送经过授权的命令；但任何会改变场景语义的操作必须回到 fps-game-editor 和项目 adapter 执行。
能力允许的平台职责必须由编辑器负责
模式切换 发送 mode:change，接收 mode:changed。 决定如何从 GameWorld 切换到 EditorWorld，以及如何释放/创建运行时对象。
文档持久化 保存、恢复、版本化编辑器导出的 opaque JSON。 解释文档 schema、校验内容、应用到编辑状态、导出运行时场景。
资产导入 提供文件、URL、sourceId、metadata 和沙盒文件能力。 决定资产如何注册、如何生成场景节点、如何进入撤销重做和脏状态。
数据更新 观察事件、转发命令、展示状态。 执行节点、材质、transform、层级、选择、历史栈等语义变更。
